</objective>

    Define `public enum CalendarPermissionStatus: Sendable` with cases: `.notDetermined`, `.fullAccess`, `.writeOnly`, `.denied`, `.restricted`. This is the protocol-level abstraction for EventKit's `EKAuthorizationStatus`. Per P-05, the consumer (adapter and UI) treats `.writeOnly` identically to `.denied`.

    Define `public struct TermRangeFilter: Sendable` with static methods:
    - `public static func filterEvents(allEvents: [CalendarEvent], recordingStart: Date, window: TimeInterval = 1800) -> [CalendarEvent]` — Swift-side filter that narrows to events overlapping `[recordingStart - window, recordingStart + window]`. This is the same ±30min logic as CourseClassifier.match but returns the raw events instead of a CourseMatch. It exists as a separate function so the EventKit adapter can pre-filter before passing events to the orchestrator, reducing the event set before CourseClassifier runs.
    - The filter uses standard interval overlap: `event.startDate <= windowEnd AND event.endDate >= windowStart`.

    The term-range predicate (CT-02) is NOT applied here in Swift — it is applied at the EventKit query level in the adapter (`predicateForEvents(withStart: term.startDate, end: term.endDate, calendars: nil)`). TermRangeFilter handles only the ±30min narrowing. This separation follows RESEARCH.md Pitfall 6: "Two-stage filter: (1) EventKit term-range predicate, (2) Swift-side ±30min filter."

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| System calendar -> app | EKEvent data crosses from EventKit into the app; event titles may contain unexpected characters |
| Permission system -> app | Authorization status may change between check and use (user revokes via Settings while app runs) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-04 | Information Disclosure | Calendar events read without Full Access | high | mitigate | fetchEvents checks authorizationStatus == .fullAccess before every query; .writeOnly throws (P-05) |
| T-04-05 | Tampering | Malicious event title in path context | medium | mitigate | Event titles are passed as CalendarEvent.title (raw string); FolderNameSanitizer strips unsafe chars before any filesystem use (existing Phase 2 T-2-01). The adapter itself does not use titles as path components. |
| T-04-06 | Denial of Service | Large calendar blocks fetchEvents on MainActor | medium | mitigate | EventKitCalendarAdapter is an actor — fetchEvents runs in actor isolation, never on MainActor. store.events(matching:) is synchronous but within the actor context, so it does not block UI. |
</threat_model>
