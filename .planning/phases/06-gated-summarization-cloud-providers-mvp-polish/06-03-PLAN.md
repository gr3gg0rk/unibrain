    Test 4: RetryComposer extracts retry-after from rateLimited error and uses that delay
    Test 5: RetryComposer throws final error after 3 failed attempts
  </behavior>
  <action>
    Create RetryComposer actor:
    - `func withRetry<T>(maxRetries: Int, operation: (Int) async throws -> T) async throws -> T`:
      1. Loop from 0..<maxRetries (3 retries per CF-03)
      2. Try await operation(attempt)
      3. On success, return result
      4. On ProviderError.rateLimited(retryAfter):
         - Use retryAfter if present, else exponential backoff: `2 << attempt` seconds (2, 8, 30)
         - await Task.sleep(nanoseconds: delay * 1_000_000_000)
         - Continue to next attempt
      5. On other ProviderError cases (.networkFailure, .modelError):
         - Use exponential backoff (2, 8, 30)
         - await Task.sleep, continue
      6. On final attempt failure, throw last error
    
    This is provider-inner retry (per CF-03). Queue-outer retry is handled separately in Phase 5 InboxQueue.
    
    Use Swift 6 Task.sleep (nanoseconds) for delay — clean async/await pattern.
    
    Create RetryComposerTests.swift:
    - Mock operation that throws specific errors on specific attempts
    - Test delay timing via clock mock or approximate verification (sleep in tests is acceptable for retry logic verification)
    - Test retry-after extraction from rateLimited error
  </action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Cloud HTTPS (api.openai.com, api.anthropic.com, docs.x.ai, api.z.ai) | HTTPS requests to cloud providers with API keys in headers. |
| APIKeyStore → Provider clients | API keys cross from Keychain to HTTP headers. |
| ConsentStore → Provider clients | Consent decisions cross from persistent file to gate logic. |
| TCPReachability → Provider clients | Network state crosses from NWConnection to gate logic. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-12 | Spoofing | Cloud HTTPS endpoints | medium | accept | ATS enforces HTTPS. Certificate pinning deferred to v2 (per 06-RESEARCH.md). Risk accepted for MVP. |
| T-06-13 | Information Disclosure | API keys in HTTP headers | low | accept | Keys in Authorization header are TLS-encrypted in transit. Acceptable for MVP. Keys never logged (verified via grep). |
| T-06-14 | Tampering | Consent check bypass | medium | mitigate | Every provider checks ConsentStore.hasConsent() BEFORE HTTP call. Unit test verifies throw on missing consent. |
| T-06-15 | Denial of Service | Retry loop without backoff | medium | mitigate | RetryComposer enforces exponential backoff (2s, 8s, 30s). Unit test verifies delay calculation. |
| T-06-16 | Tampering | TCPReachability false positive | low | accept | Reachability is pre-flight check only. False positive = user gets fallback option. No security impact. |
| T-06-17 | Information Disclosure | ProviderRouter returns wrong provider | low | mitigate | Settings snapshot cached but refreshable via updateSettings(). Unit test covers all enum cases. |
</threat_model>
