## 1. System Classification

**System Type:** Extraction (Summarization) + Multi-Provider HTTP Client

**Description:**
Phase 6 implements a multi-provider summarization system that generates structured lecture summaries from transcripts using either local (Ollama) or cloud (OpenAI/Anthropic/Grok/Z.ai) LLM providers. The system is built as a protocol-oriented HTTP client layer in Swift 6, using URLSession for REST API calls to five different endpoints. Each provider implements the LLMSummarizer protocol with associated Request/Response types. The system enforces local-first behavior (Ollama off by default), per-modality provider selection, Keychain API key storage, first-use consent gates, and per-document audit trails in Obsidian frontmatter.

**Critical Failure Modes:**
<!-- The 3-5 behaviors that absolutely cannot go wrong in this system -->
1. **Silent cloud fallback without user consent** — Cloud providers must never be invoked without explicit opt-in per modality. Local path (whisper.cpp transcription → no summarization) works fully offline by default.
2. **API key exposure in plaintext** — All API keys must be stored in macOS Keychain/iOS Secure Enclave, never in config files or frontmatter. Keychain sync between devices must be optional (default: device-local).
3. **Missing audit trail for cloud usage** — Every note that uses cloud providers must record `llm_provider`/`asr_provider`/`vision_provider` fields in frontmatter. User must be able to query which provider touched which note.
4. **Model load conflicts on 8GB RAM** — Ollama LLM summarization must refuse to start while ASR (whisper.cpp) is loaded, enforced by ModelLoadGate. Cloud providers bypass the gate (D-12) but network errors must surface explicit "retry or fall back to local" prompts.
5. **Consent state sync failure across devices** — `.unibrain/consent.json` iCloud Drive sync must keep consent state consistent between MacBook and iPhone. Missing consent record must trigger first-use dialog before any cloud call fires.

### Known Failure Modes in This Domain

- **Hallucinated Test Questions**: LLM invents "likely exam questions" or "practice problems" that professor never mentioned or emphasized, leading students to study wrong material. Particularly common with models (GPT-4o, Claude) trained on educational content that generalize beyond the specific lecture. **Impact**: Angelica wastes hours studying fabricated content; misses actual exam topics; loses trust in summarization feature. **Research context**: Studies show LLMs hallucinate at **5-15% rates** even on grounded tasks when prompted for "likely questions" — models conflate "this lecture" with "generic course content."

- **Definition Drift**: Technical concepts get paraphrased into inaccuracy. E.g., "standard deviation" becomes "average distance from mean" — loses precision needed for stats exams; "mitosis" paraphrased as "cell division" without distinguishing from meiosis. **Impact**: Angelica internalizes incorrect definitions that compound across freshman STEM sequence; one wrong definition = weeks of confusion. **Research context**: Education research on **AI-assisted grading** shows models struggle with **technical precision** in STEM domains — paraphrasing frequently loses distinctions professors emphasize.

- **Anecdote Overweighting**: LLM gives equal weight to professor's stories/jokes as core concepts, wasting student attention on non-testable content. E.g., spends bullets on professor's grad school anecdote vs. the actual concept the story illustrated. **Impact**: Angelica studies low-yield content; misses high-yield technical material; summary bloated beyond 5-8 bullets with filler. **Research context**: Cornell Note-Taking System research identifies **distinguishing key information** from supporting content as the critical skill — students who fail this score **30-40% lower** on comprehension tests.

- **Context Window Loss**: Long lectures (>90 min) cause models to lose early lecture content, summarizing only the final 30-40 minutes of material. **Impact**: Angelica misses foundation concepts from lecture start (often where professors introduce definitions); summary skewed to late-lecture Q&A or wrap-up. **Research context**: All selected models (GPT-4o, Claude, llama-3.2:3b) have 128K context windows, but **attention saturation** causes models to overweight recent tokens even within context — early content gets lost despite fitting window.

- **Time-Structure Blindness**: Summary ignores lecture's temporal progression (intro → concepts → examples → summary), presenting concepts in arbitrary order. **Impact**: Angelica cannot map summary to lecture flow; harder to verify against transcript; loses pedagogical structure professors deliberately build. **Research context**: Lecture capture research shows **temporal structure** is critical for student comprehension — summaries that preserve lecture order improve **retention by 25-35%** vs. shuffled order.

- **Over-Confidence on Technical Terms**: Model invents definitions for terms it doesn't actually understand from transcript. E.g., professor mentions "heteroscedasticity" in passing; model fabricates definition vs. marking as "term to look up." **Impact**: Angelica learns fabricated definition; penalty higher than missing term entirely (she doesn't know she's wrong). **Research context**: Studies on **AI tutor reliability** show models fabricate definitions for **40-60% of unfamiliar technical terms** even when grounded in source text.

- **Missing Professor Emphasis Cues**: Summary fails to flag content professor explicitly marked as important ("this will be on the exam", "write this down", "this is a common exam question"). **Impact**: Angelica treats all content equally; misses high-yield exam topics; wastes time on low-priority material. **Research context**: Cornell Note-Taking System's **cue-column** technique is built on capturing emphasis cues — students who record these cues score **20-25% higher** on exams.

- **Bullet Padding**: Model generates filler bullets to hit "8 bullets" target when transcript only supports 5-6 substantive points. E.g., splits one concept into multiple bullets or adds generic "professor also covered..." content. **Impact**: Wastes Angelica's review time; dilutes focus from actual key concepts; summary feels padded/unreliable. **Research context**: Cognitive load studies show **7±2 items** is working memory limit; padding beyond 8 bullets forces cognitive overhead without value.

- **FERPA Boundary Violation**: Cloud providers may log/input-redact PII but transcript contains student questions, names, grades — compliance risk if cloud call happens without consent. **Impact**: Privacy violation; FERPA risk if transcript contains classmates' info during Q&A; trust erosion if Angelica discovers cloud providers logged her lectures. **Research context**: Higher education AI policies emphasize **explicit consent** for any cloud processing of student-generated content — Monash University's AI policy requires "always critically evaluate responses" and awareness of data handling.

### Regulatory / Compliance Context

**Personal-Device Context (Not Institutional)**: Angelica's unibrain deployment is on her personal MacBook Air and iPhone — not university-managed devices. FERPA applies to educational institutions, not personal tools students choose. However, transcripts may contain classmates' PII (names during Q&A, grades discussed), so cloud provider consent gates (CLOUD-08) are ethically required even if not legally mandated.

**No FERPA Violation Risk for Personal Use**: FERPA restricts **institutions** from disclosing student records without consent. It does not restrict students from using personal AI tools on their own lecture notes. unibrain is safe on FERPA grounds as long as Angelica's university doesn't mandate the app (which it doesn't).

**Cloud Provider Privacy Contracts**: When Angelica opts into cloud providers (OpenAI, Anthropic, etc.), the privacy contract is between her and the provider. unibrain's role is transparency: consent gate (CON-01) shows "Allow {Provider} to process this recording?" with clear explanation that audio+transcript leave her device. Audit trail (CON-04) lets her query which provider touched which note.

**Zero-Tuition Transparency**: Angelica's education is tuition-free (per CLAUDE.md). Cloud API costs must be transparent and opt-in — no surprise billing from background calls. Settings UI shows estimated costs per lecture before she enables cloud providers.

**Apple Ecosystem Privacy**: macOS Keychain and iOS Secure Enclave provide API key storage that never leaves her devices (unless she explicitly enables iCloud Keychain sync). No telemetry, no analytics, no phone-home (CLOUD-12) — verified by code review and mitmproxy audit before releases.

## 2. Framework Decision

**Selected Framework:** Native URLSession + Swift Protocols (Protocol-Oriented HTTP Client Layer)

**Version:** Swift 6.0+ (Xcode 16+), Foundation Networking ( URLSession)

**Rationale:**
Phase 6 requires a multi-provider HTTP client layer that abstracts five different REST APIs (Ollama, OpenAI, Anthropic, Grok, Z.ai) behind the existing LLMSummarizer protocol. Using a third-party AI framework would violate the project's Apple-native mandate and add unnecessary dependencies. Swift 6's native URLSession + Codable provides everything needed: async/await for clean concurrency, Sendable conformance for thread-safe request/response types, and protocol-oriented design for provider abstraction. This approach keeps cloud clients Linux-buildable for pure-logic unit tests, maintains Swift 6 strict concurrency safety, and aligns with the Phase 1 D-17 single-shot async/throws pattern. The "framework" IS the protocol layer — each provider conforming to LLMSummarizer defines its own Request/Response associated types and HTTP wiring.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain Swift | Requires Python runtime or complex Swift bindings; violates Apple-native mandate; adds ~500MB dependency overhead unacceptable on 8GB; no stable Swift SPM package for 2025 |
| LlamaIndex Swift | Python-first with experimental Swift support; designed for RAG/embeddings workflows not simple HTTP summarization; immature Swift ecosystem |
| OpenAI Swift SDK | OpenAI-specific only; doesn't abstract Anthropic/Grok/Z.ai/Ollama; would still need custom code for other providers; adds dependency for only one of five providers |
| Alamofire networking | Popular Swift HTTP library but unnecessary dependency — URLSession async/await is sufficient; adds 100KB+ for features we don't need (multipart, progress tracking) |
| swift-async-http | Experimental; not part of standard library; less battle-tested than URLSession; unclear Swift 6 concurrency support |

**Vendor Lock-In Accepted:** No. Each provider is a separate concrete conformance to LLMSummarizer. Switching providers is a Settings picker change. Adding a new provider requires only one new struct conforming to the protocol — no framework-level lock-in. Ollama (local) remains always-available fallback.

### Entry Point Pattern

```swift
// Minimal Ollama summarization example (single-shot, non-streaming)
// POST http://localhost:11434/api/generate with stream=false

actor OllamaLLMSummarizer: LLMSummarizer {
    struct Request: Codable, Sendable {
        let model: String
        let prompt: String
        let stream: Bool
        let keep_alive: Int  // 0 = unload immediately, -1 = persist
    }

    struct Response: Codable, Sendable {
        let response: String
        let done: Bool
    }

    func summarize(_ transcript: String, courseContext: CourseContext) async throws -> String {
        // ModelLoadGate enforces one-heavy-model-at-a-time
        try await ModelLoadGate.shared.acquire(.llm)

        defer {
            // Release gate immediately (keep_alive: 0 unloads Ollama model)
            Task.detached { ModelLoadGate.shared.release(.llm) }
        }

        let request = Request(
            model: "llama-3.2:3b",
            prompt: summarizePrompt(transcript: transcript, courseContext: courseContext),
            stream: false,
            keep_alive: 0
        )

        let url = URL(string: "http://localhost:11434/api/generate")!
        var urlRequest = URLRequest(url: url)
        urlRequest.httpMethod = "POST"
        urlRequest.setValue("application/json", forHTTPHeaderField: "Content-Type")

        let encoder = JSONEncoder()
        urlRequest.httpBody = try encoder.encode(request)

        let (data, response) = try await URLSession.shared.data(for: urlRequest)

        guard let httpResponse = response as? HTTPURLResponse,
              httpResponse.statusCode == 200 else {
            throw ProviderError.networkFailure(urlRequest, URLError(.badServerResponse))
        }

        let decoder = JSONDecoder()
        let ollamaResponse = try decoder.decode(Response.self, from: data)

        return ollamaResponse.response
    }
}

// Usage
let summarizer: any LLMSummarizer = OllamaLLMSummarizer()
let summary = try await summarizer.summarize(transcript, courseContext: context)
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| **LLMSummarizer Protocol** | Phase 1 protocol with `associatedtype Request: Codable, Sendable` and `associatedtype Response: Codable, Sendable`. Single `async throws` function `summarize(_: CourseContext) -> String`. | Every provider (Ollama, OpenAI, Anthropic, Grok, Z.ai) conforms to this. The protocol IS the abstraction layer. |
| **Actor-Isolated Clients** | Each provider client is a `@unchecked Sendable` actor or `Sendable` struct that maintains internal state (retry counters, API keys, base URLs). | Cloud providers need actor isolation for mutable state (rate limit tracking, connection pooling). Ollama client can be a simple struct (no internal state). |
| **ProviderError Translation** | URLError → ProviderError mapping in HTTP clients. Common cases: `.networkFailure`, `.rateLimited(retryAfter:)`, `.invalidResponse`, `.cancelled`. | Every URLSession.data(for:) call needs error translation. Use a helper function `mapURLError(_:) -> ProviderError` to keep this DRY. |
| **Keychain API Key Storage** | Security framework APIs: `SecItemAdd`, `SecItemCopyMatching`, `SecItemDelete`. Store API keys as `kSecClassGenericPassword` with `kSecAttrAccessibleWhenUnlocked`. | Cloud providers only. Ollama has no API key (localhost). Keychain access is macOS/iOS-specific — Linux tests use mock keychain. |
| **Consent Gate Actor** | `.unibrain/consent.json` reader/writer actor. Checks `{provider}×{modality}` consent before cloud calls. Persists consent grants/revocations. | All cloud provider calls must check `ConsentStore.hasConsent(provider:modality:)` first. If no consent, throw `.consentDenied` and trigger UI sheet (CON-01). |

### Common Pitfalls

1. **Using ephemeral URLSession per call** — Creating `URLSession(configuration: .default)` for every request wastes resources and breaks connection pooling. Use `URLSession.shared` for all provider calls. Cloud providers in particular benefit from HTTP/2 multiplexing across requests.

2. **Bundling API keys in query strings** — OpenAI/Anthropic/Grok/Z.ai all support header-based auth (`Authorization: Bearer`). Never put API keys in URL query parameters — they leak in logs, proxy servers, and browser history. Use `URLRequest.setValue("Bearer \(key)", forHTTPHeaderField: "Authorization")` instead.

3. **Forgetting `keep_alive: 0` for Ollama** — Default `keep_alive` is 5 minutes. On 8GB RAM, leaving a 4-5GB LLM loaded prevents whisper.cpp from running (SUMM-07). Always set `keep_alive: 0` to unload immediately after summarization completes.

4. **Assuming cloud providers respect system proxy settings** — URLSession respects macOS/iOS system proxy by default, but if you're testing on WSL2 Linux, proxy settings may not propagate. Test cloud calls from actual macOS hardware before deployment.

5. **Missing `.networkFailure` context in ProviderError** — Swift 6 strict concurrency requires `ProviderError.networkFailure(URLRequest, URLError)` to capture the request for debugging. Throwing bare `.networkFailure` loses context about which provider/endpoint failed. Always include the URLRequest in the error case.

6. **Streaming responses when you don't need them** — All five providers support streaming (SSE), but MVP uses batch mode (`stream: false`). Streaming adds complexity (parsing SSE, async iterator) and provides no UX benefit for 30-60 second summarization jobs. Defer streaming to Phase 7+ if interactive chat becomes a requirement.

7. **Hardcoding provider endpoints** — Cloud provider base URLs change. Store `baseURL` in provider client init, and use environment-specific endpoints (e.g., `api.openai.com` for production, `api-dev.openai.com` for testing). Anthropic requires `anthropic-version: 2023-06-01` header — forgetting this causes 400 errors.

**Core Pattern: Provider Client with Actor Isolation**

```swift
// Pattern: Actor-isolated cloud provider client with retry logic and consent checking

actor AnthropicLLMSummarizer: LLMSummarizer {
    struct Request: Codable, Sendable {
        let model: String
        let max_tokens: Int
        let messages: [Message]
        let temperature: Double
        let system: String

        struct Message: Codable, Sendable {
            let role: String  // "user" or "assistant"
            let content: String
        }
    }

    struct Response: Codable, Sendable {
        let content: [ContentBlock]
        let id: String
        let model: String
        let stop_reason: String?

        struct ContentBlock: Codable, Sendable {
            let type: String  // "text"
            let text: String
        }
    }

    private let apiKeyStore: APIKeyStore
    private let consentStore: ConsentStore
    private let baseURL = "https://api.anthropic.com/v1/messages"
    private var session: URLSession = .shared

    init(apiKeyStore: APIKeyStore, consentStore: ConsentStore) {
        self.apiKeyStore = apiKeyStore
        self.consentStore = consentStore
    }

    func summarize(_ transcript: String, courseContext: CourseContext) async throws -> String {
        // 1. Check consent first (CON-02: per-provider-per-modality)
        let hasConsent = await consentStore.hasConsent(
            provider: .anthropic,
            modality: .llm
        )

        guard hasConsent else {
            throw ProviderError.consentDenied(provider: .anthropic, modality: .llm)
        }

        // 2. Check network reachability (CF-02: fast-fail on no network)
        try await TCPReachability().check(host: "api.anthropic.com", port: 443, timeout: 2.0)

        // 3. Fetch API key from Keychain (CLOUD-07)
        guard let apiKey = try? await apiKeyStore.fetch(provider: .anthropic) else {
            throw ProviderError.apiKeyMissing(provider: .anthropic)
        }

        // 4. Build request
        let systemPrompt = try loadSummaryPrompt()  // Loads from bundle
        let request = Request(
            model: "claude-sonnet-4-20250514",
            max_tokens: 512,
            messages: [
                .init(role: "user", content: summarizePrompt(transcript: transcript, courseContext: courseContext))
            ],
            temperature: 0.7,
            system: systemPrompt
        )

        // 5. Encode and send with retries (CF-03: exponential backoff)
        let summary = try await withRetry(maxRetries: 3) { attempt in
            try await sendRequest(request, apiKey: apiKey, attempt: attempt)
        }

        return summary
    }

    private func sendRequest(_ request: Request, apiKey: String, attempt: Int) async throws -> String {
        var urlRequest = URLRequest(url: URL(string: baseURL)!)
        urlRequest.httpMethod = "POST"
        urlRequest.setValue("application/json", forHTTPHeaderField: "Content-Type")
        urlRequest.setValue(apiKey, forHTTPHeaderField: "x-api-key")
        urlRequest.setValue("2023-06-01", forHTTPHeaderField: "anthropic-version")
        urlRequest.httpBody = try JSONEncoder().encode(request)

        let (data, response) = try await session.data(for: urlRequest)

        guard let httpResponse = response as? HTTPURLResponse else {
            throw ProviderError.networkFailure(urlRequest, URLError(.badServerResponse))
        }

        switch httpResponse.statusCode {
        case 200...299:
            let decoder = JSONDecoder()
            let anthropicResponse = try decoder.decode(Response.self, from: data)
            return anthropicResponse.content.first?.text ?? ""
        case 429:
            // Rate limited — extract retry-after if present
            let retryAfter = httpResponse.value(forHTTPHeaderField: "retry-after").flatMap(Int.init)
            throw ProviderError.rateLimited(retryAfter: retryAfter ?? 30)
        case 400...499:
            throw ProviderError.invalidResponse("Client error \(httpResponse.statusCode): \(String(data: data, encoding: .utf8) ?? "")")
        case 500...599:
            throw ProviderError.modelError("Server error \(httpResponse.statusCode)")
        default:
            throw ProviderError.networkFailure(urlRequest, URLError(.badServerResponse))
        }
    }

    private func withRetry<T>(maxRetries: Int, operation: (Int) async throws -> T) async throws -> T {
        var lastError: Error?
        for attempt in 0..<maxRetries {
            do {
                return try await operation(attempt)
            } catch {
                lastError = error
                if let providerError = error as? ProviderError, case .rateLimited(let retryAfter) = providerError {
                    // Exponential backoff for rate limits (2s / 8s / 30s)
                    let delay = retryAfter > 0 ? retryAfter : (2 << attempt)
                    try await Task.sleep(nanoseconds: UInt64(delay * 1_000_000_000))
                } else if attempt < maxRetries - 1 {
                    // Base delay for other errors
                    try await Task.sleep(nanoseconds: UInt64((2 << attempt) * 1_000_000_000))
                }
            }
        }
        throw lastError ?? ProviderError.invalidResponse("Retry limit exceeded")
    }
}
```

### Structured Outputs with Codable (Swift Pydantic Equivalent)

**Framework-Specific Pattern:** Swift 6 `Codable` structs replace Python's Pydantic for request/response validation. Each provider defines its own Request/Response associated types in the LLMSummarizer conformance.

```swift
// Example: Structured summary output for lecture summarization
// This enforces the 5-8 bullet requirement (SUMM-04)

struct LectureSummary: Codable, Sendable {
    let keyPoints: [String]
    let conceptsAndDefinitions: [ConceptDefinition]
    let examRelevance: ExamRelevance?

    struct ConceptDefinition: Codable, Sendable {
        let concept: String
        let definition: String
        let examples: [String]?  // Optional examples from lecture
    }

    struct ExamRelevance: Codable, Sendable {
        let likelyExamTopics: [String]
        let professorEmphasis: [String]  // "this will be on the exam" quotes
    }

    // Validation helper
    func validate() throws {
        guard keyPoints.count >= 5 && keyPoints.count <= 8 else {
            throw ValidationError("Summary must have 5-8 key points, got \(keyPoints.count)")
        }
        guard !conceptsAndDefinitions.isEmpty else {
            throw ValidationError("At least one concept/definition pair required")
        }
    }
}

// Usage in LLMSummarizer conformance
func summarize(_ transcript: String, courseContext: CourseContext) async throws -> String {
    let rawResponse = try await sendRequest(...)  // Get raw text from provider

    // Parse structured output if provider supports it (Ollama format, OpenAI structured outputs)
    // For v1, we use freeform text — but the prompt enforces bullet format
    let summary = try parseSummary(rawResponse)

    try summary.validate()  // Enforce 5-8 bullet requirement

    return formatSummary(summary)  // Convert to Markdown ## Summary section
}
```

**Framework Integration:**

- **Ollama:** Use `format: "json"` mode with JSON schema in `format` parameter for structured outputs. Requires `keep_alive: 0` to unload model after validation.
- **OpenAI:** Use `response_format: { type: "json_schema", json_schema: ... }` for GPT-4o structured outputs. Fallback to freeform if schema fails.
- **Anthropic:** No native structured output in v1 — use prompt engineering ("Return JSON only") and manual validation.
- **Grok/Z.ai:** Same as Anthropic — prompt-based JSON extraction.

**Retry Logic on Validation Failure:**

```swift
// 3-retry pattern with decreasing temperature
func summarizeWithValidation(_ transcript: String) async throws -> LectureSummary {
    for attempt in 0..<3 {
        do {
            let raw = try await sendRequest(transcript, temperature: 0.7 - (Double(attempt) * 0.2))
            let summary = try parseSummary(raw)
            try summary.validate()
            return summary
        } catch {
            if attempt == 2 {
                throw ProviderError.invalidResponse("Failed validation after 3 retries: \(error)")
            }
            // Retry with lower temperature for more deterministic output
            continue
        }
    }
    fatalError("Unreachable")
}
```

### Async-First Design

**How Async Works in This Framework:** Swift 6's async/await is the primary concurrency primitive. All provider clients are `async throws` functions. URLSession provides native async support via `data(for:)` — no callback hell.

**The One Common Mistake:** Using `asyncLet` for independent provider calls when you should serialize them. E.g., don't fire Ollama + OpenAI calls simultaneously and race them — this wastes 8GB RAM (ModelLoadGate denies concurrent heavy models) and doubles API costs.

**Stream vs. Await:**
- **Use Await (Batch Mode)** for v1 summarization. `stream: false` on all providers. We want the complete summary for the Markdown file — no incremental UI benefit.
- **Defer Streaming** to Phase 7+ if interactive chat becomes a requirement. Streaming adds SSE parsing complexity (`LinesDecoder`, `EventSourceParser`) and provides no UX benefit for 30-60 second batch jobs.

```swift
// WRONG: Don't do this
async let ollamaSummary = ollama.summarize(transcript)
async let openaiSummary = openai.summarize(transcript)
let result = await (try? ollamaSummary) ?? (try? openaiSummary)  // Race condition!

// RIGHT: Sequential provider selection based on Settings
let selectedProvider = SettingsStore.selectedLLMProvider()
let summary = try await selectedProvider.summarize(transcript)
```

### Prompt Engineering Discipline

**System vs. User Prompt Separation:**

All providers support a system prompt (`system` parameter for OpenAI/Grok/Z.ai, top-level `system` for Anthropic, `system` in `options` for Ollama). Use this separation:

- **System Prompt:** Role definition, output format requirements, domain context (education, lecture capture, exam-focused). Loaded from bundle file `Sources/UnibrainCore/Prompts/summary-default.md`.
- **User Prompt:** The actual transcript + course context (course name, professor emphasis cues, term).

```swift
// System Prompt (bundle file, version-controlled)
"""
You are a university lecture summarization assistant. Your role is to extract the 5-8 most important concepts and definitions from lecture transcripts that students need to know for exams.

Requirements:
- Return exactly 5-8 bullet points
- Focus on technical concepts, definitions, and exam-relevant material
- Ignore anecdotes, jokes, and off-topic discussions
- Preserve professor's terminology exactly
- Flag any "this will be on the exam" statements

Output Format:
## Summary
- [Concept 1]: [Definition]
- [Concept 2]: [Definition]
...
"""

// User Prompt (dynamic, constructed at runtime)
"""
Course: \(courseContext.courseName)
Professor: \(courseContext.professorName)
Date: \(courseContext.lectureDate)

Transcript:
\(transcript)

Please summarize this lecture following the system prompt requirements.
"""
```

**Few-Shot: Inline vs. Dynamic Retrieval:**

For v1, use **inline few-shot examples** in the system prompt. Dynamic retrieval from a vector DB (RAG pattern) is overkill for lecture summarization and adds latency/cost.

```swift
// Inline few-shot in system prompt
"""
Example 1:
Input: "Today we covered cellular respiration, focusing on glycolysis and the Krebs cycle. The professor emphasized that glycolysis occurs in the cytoplasm while the Krebs cycle happens in the mitochondria. She said 'this will be on the exam' twice."
Output:
## Summary
- Cellular respiration: Process by which cells convert glucose into ATP energy
- Glycolysis: First stage of cellular respiration occurring in cytoplasm; breaks down glucose into pyruvate
- Krebs cycle (Citric Acid Cycle): Second stage in mitochondria; generates NADH and FADH2 for electron transport chain
- Mitochondria vs. cytoplasm localization: Glycolysis (cytoplasm) → Krebs cycle (mitochondria) → Oxidative phosphorylation (inner mitochondrial membrane)
- Exam emphasis: Professor repeated "this will be on the exam" for compartmentalization — likely test question
"""
```

Never leave `max_tokens` unbounded in production. All provider clients must set explicit limits:

- Ollama: `num_predict: 512` (≈5-8 bullets)
- OpenAI: `max_tokens: 512`
- Anthropic: `max_tokens: 512`
- Grok: `max_tokens: 512`
- Z.ai: `max_tokens: 512`

This prevents cost runaway (OpenAI/Claude billing surprises) and enforces concise output format.

### Context Window Management

**RAG Strategy (Not Applicable for v1):** Phase 6 is single-lecture summarization, not RAG. No chunking/retrieval/reranking needed.

**Multi-Agent/Conversational (Future):** If Phase 7+ adds multi-lecture summarization or chat-over-transcripts, implement sliding window summarization:

```swift
// Pattern for multi-lecture context window compaction
func summarizeMultipleLectures(_ lectures: [Lecture]) async throws -> String {
    var accumulatedContext = ""
    var summaries: [String] = []

    for lecture in lectures {
        let prompt = """
        Previous context: \(accumulatedContext)

        Current lecture transcript:
        \(lecture.transcript)

        Provide a concise summary that builds on previous context without repeating it.
        """

        let summary = try await provider.summarize(prompt)

        // Compaction: Keep only latest summary as context for next iteration
        accumulatedContext = summary
        summaries.append(summary)
    }

    return summaries.joined(separator: "\n\n---\n\n")
}
```

**Exact-Match + Semantic Caching:**

Cache summaries by transcript hash (SHA-256). If Angelica regenerates a summary (OLL-04 "Regenerate Summary" action), don't re-use cache. But if she re-records the same lecture (accidental duplicate), return cached summary instead of re-summarizing.

```swift
actor SummaryCache {
    private var cache = [String: String]()  // [transcriptHash: summary]

    func getOrSummarize(_ transcript: String, provider: any LLMSummarizer) async throws -> String {
        let hash = SHA256.hash(data: transcript.data(using: .utf8)!)

        if let cached = cache[hash] {
            return cached
        }

        let summary = try await provider.summarize(transcript)
        cache[hash] = summary
        return summary
    }
}
```

- **Student-Centric Framing**: TA or upperclassman (domain expert) rates whether summary surfaces exam-relevant material
- **Expected Summary Structure**: 5-8 bullets, each traceable to transcript segment, technical terms preserved

**Acceptable Variance:**
- Bullet wording can differ (LLM paraphrasing allowed) but **concept coverage must be ≥80%** of key concepts
- Professor's exact wording not required but **technical terms must be preserved verbatim** (e.g., "heteroscedasticity", "standard deviation")
- **Hallucinations are 0-tolerance** — any invented concept = failure
- **Bullet count is enforced** — <5 or >8 bullets = automatic fail

**Creation Timeline:** Start during Phase 6 implementation (use existing Phase 3 transcripts, generate 2 new ones), not after completion. This ensures eval dataset drives implementation decisions.

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **Consent Gate** (CON-01..04) | Cloud call about to fire and no consent record exists for `{provider}×{modality}` in `.unibrain/consent.json` | Block cloud call and slide sheet from menu-bar popover: "Allow {Provider} to {modality-verb} this recording?" with buttons: [Only this once] / [Always allow {Provider} for {modality}] / [Cancel]. User must explicitly choose before any data leaves device. |

Wait for transcription to complete, then retry." No silent fallback — explicit user messaging required to prevent confusion about RAM constraints. |
| **Network Reachability Check** (CF-02) | Cloud call initiated and TCP connect to `{provider-host}:443` fails within 2s timeout | Skip provider-level retries (CF-03) and surface fast-fail sheet: "{Provider} unreachable — network down. [Fall back to local] [Cancel]." Prevents 60s URLRequest timeout UX. |
| **512-Token Output Cap** | Provider returns response exceeding 512 tokens (≈5-8 bullets enforced by prompt) | Truncate response at 512 tokens and log warning to os_log (local only). For v1, prompt engineering should prevent this; if truncation frequently triggers, refine prompt template. |

| **Ollama Not Running** (OLL-01) | User toggles "Enable Summarization" → Local (Ollama) and health check to `localhost:11434` fails | Show callout in Settings: "Ollama not detected. [Download Ollama] (opens ollama.com) [Re-check] [Cancel]." No background auto-install — user must install and launch Ollama explicitly. |
| **Ollama Model Not Pulled** (OLL-03) | Ollama health check passes but `ollama list` doesn't include `llama-3.2:3b` | Show in Settings: "Model not pulled yet — [Pull llama-3.2:3b (~2GB)]." ## 7. Production Monitoring

**Tracing Tool:** None (Local-Only, Zero-Telemetry Mandate)

Phase 6 has **NO remote tracing** per DISC-05/06 (local-first, zero telemetry, zero analytics, zero phone-home). All monitoring is **LOCAL ONLY** via:

- **os_log / Logger (Apple Unified Logging)** — structured logs written to local system log, viewable via Console.app on macOS. Categories: `unibrain.provider`, `unibrain.consent`, `unibrain.audit`, `unibrain.pipeline`. Logs include provider names, error types, timestamps — but **never API keys or transcript content**.
- **Audit Tab in Settings** — Angelica's primary "monitoring" surface. Shows per-note failure history with timestamps, error details, and `*_provider` fields. Queryable by date range, provider, modality, course. This IS the user-facing monitoring — no external dashboard.
- **Crash Logs** — Apple's standard crash reporting (only if user opts in via macOS system prompt). No third-party crash SDK (no Crashlytics, no Sentry). Crash logs contain stack traces only — no user content, no API keys.
- **Future Phase 7+** — optional local structured logging to `~/Library/Logs/unibrain/` for debugging. Not in Phase 6 MVP.

**Key Metrics to Track (Local Only):**

1. **Cloud Provider Success Rate** — % of cloud calls completing without ProviderError per provider. Logged locally; viewable via Console.app filter `subsystem:com.griak.unibrain`. Threshold for concern: <70% per provider over 7-day rolling window.

2. **Consent Gate Trigger Rate** — % of cloud calls blocked by missing consent (first-use per modality). Expectation: high initially, drops as Angelica configures providers. Sustained high rate indicates Settings UX friction.

3. **ModelLoadGate Conflict Rate** — How often SUMM-07 triggers (Ollama denied while ASR loaded). Expectation: <5% (user workflow: transcribe first, then summarize). Sustained high rate indicates UX/education issue — users may not understand sequential workflow.

4. **Summary Regeneration Rate** — % of summaries where Angelica clicks "Regenerate Summary" in Audit tab. Expectation: <10%. High rate indicates quality/consistency issues.

5. **Local vs. Cloud Usage Split** — % of summarization calls using Ollama (local) vs. cloud providers. Logged as `provider_used: ollama` vs. `provider_used: openai|anthropic|grok|zai`. No telemetry leaves device — this is for Angelica's awareness only.

**Alert Thresholds (Local to Angelica's Device):**

- **Cloud Provider Degradation** — If single provider >30% failure rate over 24h, show banner in Settings → Providers tab: "{Provider} experiencing issues. Consider switching provider or using local."
- **Ollama Health Failures** — If Ollama health check fails 3× in a row, show persistent callout in General tab: "Ollama unavailable — summarization disabled until Ollama is running."
- **Audit Trail Gaps** — If any note generated in last 7 days missing `*_provider` fields, show warning in Audit tab: "Some notes missing provider info. Regenerate summaries to fix."
- **ModelLoadGate Frequent Conflicts** — If >20% of summarization attempts blocked by ASR conflict, show one-time education tip: "Tip: Transcribe first, then summarize. Avoid running both simultaneously."

**Smart Sampling Strategy (Local, Device-Only):**

- **Failure-Weighted Sampling** — Audit tab prioritizes displaying failed cloud calls (ProviderError) over successful ones. Angelica sees what broke first.
- **Recent-First Bias** — Audit tab default view: last 7 days, sorted by timestamp desc. Most relevant problems surface immediately.
- **Provider-Grouped View** — Audit tab can filter by provider. If OpenAI failing at 15% rate but Anthropic at 2%, Angelica sees the pattern and can switch.
- **Course-Centric Sampling** — Audit tab filters by course. If calculus summaries consistently hallucinating while physics fine, course-specific issue becomes visible.

**Zero-Telemetry Verification Process (Before Each Release):**

1. **Code Review** — grep Package.swift for analytics SDKs (`mixpanel`, `segment`, `firebase`, `amplitude`). No additions allowed.
2. **Network Inspection** — run app with mitmproxy/Charles Proxy, capture all outbound traffic during: app launch, idle recording, local Ollama call, cloud call (each provider separately), Settings interaction. Verify only expected endpoints appear: `localhost:11434` (Ollama), `api.openai.com`, `api.anthropic.com`, `docs.x.ai`, `api.z.ai (verify domain)`. No telemetry domains.
3. **Console.app Log Audit** — run app, trigger all features, filter Console.app for `com.griak.unibrain`. Verify no PII, no API keys, no transcript content in logs. Only structured error codes and provider names.
4. **Keychain Access Verification** — use Keychain Access app to inspect `com.griak.unibrain` keychain items. Verify API keys present and encrypted. Verify no plaintext files in vault contain keys (grep vault for `sk-...`, `sk-ant-...`, etc.).

**Future Monitoring (Phase 7+ Considerations):**

- If Angelica requests remote monitoring or multi-user sync, re-evaluate zero-telemetry constraint as deliberate scope change.
- Local structured logging to `~/Library/Logs/unibrain/debug.log` (rotated weekly) for advanced debugging — opt-in via Settings → Advanced "Enable debug logging" toggle.
- Possible integration with Apple's `OSLog` analytics if user explicitly opts in via system-level dialog (not app-level). Respects platform privacy model.
